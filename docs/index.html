Attempted to add a pinning test for the `already_done` comment but discovered the code path is architecturally unreachable: if a Done board item links to an issue number, that issue is caught by the `alreadyOnBoard` pre-filter before reaching LLM triage, so the Done-gate check for `already_done` can never pass. Learnings

[anti-pattern] Adding test assertions for values populated via side-effecting mocks without re-checking that the relevant mock state survives `vi.clearAllMocks()` inter-test cleanup — solution is to assert only what the specific test is exercising. [domain] In the triage pipeline, the `alreadyOnBoard` filter and the Done-gate guard are mutually exclusive by design: any issue linked to a Done board item is filtered out before LLM triage, so the `already_done` LLM action can never survive the Done-gate via the normal flow. [pattern] When making a sync function async, all callers must be updated to `await` it — missing `await` silently swallows errors and leaves async operations unfinished during tests. [domain] Closing GitHub issues only upon confirmed completion (Done-transition) rather than at triage time preserves the community signal: an open issue means in-progress, a closed issue means resolved with evidence.
